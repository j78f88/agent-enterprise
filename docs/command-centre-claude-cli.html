info Invocation. The original Command Centre uses @agent (Copilot) shorthand. In Claude Code the same 13 agents are slash commands you type at the prompt: /planner, /sprint-lead, and so on. Inside a sprint they run as subagents, each in its own context handing back structured JSON. The agents behave the same way whichever runtime you're on; what changes is how you call them.

smart_toy The roster — as slash commands
Type these at the prompt. The strategy → planning → delivery chain is a run of slash commands you trigger yourself. After that, /sprint-lead calls the quality and docs lane as subagents, so you don't have to invoke them one by one.
Strategywhat & why & how — opt-in, never implements

/researcher subagent

External research. Digs up patterns, data models, user complaints, and failure modes from outside the codebase. Reports what it finds and leaves the call to you.
/researcher <research question>
→ hands off to /pm · writes research docs

/pm subagent

Product manager. Validates whether a feature is worth building (5-test echo-chamber filter). Owns the why and when. Never writes sprint plans.
/pm should we build <idea>
/pm validate this feature
→ hands off to /planner · writes validation records

/architect subagent

Technical advisor. Designs approaches, writes ADRs, pressure-tests before planning. Prefers the boring option. Never implements code.
/architect design <approach>
/architect write an ADR for <decision>
→ hands off to /planner · writes ADR + architecture docs
Planning & Captureintent → plan · drafts/ → sprints/

/planner subagent

Business analyst & sprint planner. Scopes requirements, drafts sprint plans, manages backlog ledger and non-goals. Drafts in drafts/, promotes to sprints/. Has a draft-approval checkpoint — you approve before it commits.
/planner scope this work
/planner draft a sprint
/planner check the backlog
→ hands off to /sprint-lead · rejects back to /pm · /architect

/bug

Fast bug capture (~30s). Appends to the bug backlog with ID, severity, area, repro steps. Never diagnoses or plans fixes.
/bug <describe the bug>
→ hands off to /planner for fix planning · writes backlog entries
Delivery6-phase sprint orchestration

/sprint-lead orchestrator

Thin orchestrator. Reads PLAN.md, hands every task to its own subagent, runs the quality gates, gathers the results, and writes the sprint report and retro. It never writes code itself.
/sprint-lead run the sprint
/sprint-lead run Sprint N autopilot
/sprint-lead continue Sprint N
↳ invokes /qa · /a11y · /perf · /security · /reviewer · /docs as subagents
Quality Gatesread-only — report findings, never modify code

/qa

Full quality pipeline: typecheck, lint, unit tests, coverage, E2E. Reports pass/fail with exact numbers vs configured thresholds.
/qa run tests · /qa validate sprint
read-only — reports only

/reviewer

Code review for pattern compliance, type quality, security basics, planning-artifact integrity. Primary severity classifier.
/reviewer review this code
/reviewer review branch <name>
read-only — reports only

/security

Vulnerabilities, active exploits, security changelog, file-integrity hashes. Cites CVE IDs and advisory URLs.
/security security audit
/security CVE scan · /security integrity check
reports + writes security docs

/perf

Bundle size, build time, dependency health. Flags unused / missing / duplicated deps vs configured budgets.
/perf performance audit · /perf dependency audit
read-only — reports only

/a11y

WCAG 2.1 AA audit: keyboard nav, ARIA semantics, contrast, screen-reader behaviour (automated + manual).
/a11y accessibility audit <scope>
read-only — reports only
Documentation & Setuppost-sprint sync · only documents what exists

/docs

Maintains all project docs after a sprint: SPRINTS.md, changelog, user guide, architecture, dev docs. Never invents features.
/docs documentation sync · /docs write release notes
invoked by /sprint-lead in Phase 5 · writes doc files

/onboarding setup-only

First-time setup for a new project: profile selection, config fill, token resolution, verification. Self-removes when done. Your first stop on a fresh checkout.
/onboarding get started · /onboarding set up agents
lifecycle: self-removes after completion
timeline The sprint flow
PRIMARY CHAIN /researcher → /pm → /planner → /sprint-lead → [gates] → /docs │ │ writes PLAN.md spawns 1 subagent per task
What /sprint-lead runs for you, in order. Kick it off, then either answer the EXIT POINT prompts as they come (interactive) or let it run start to finish (autopilot).
PHASE 1

Kickoff

Reads PLAN.md, validates every task has a Files: annotation, builds the todo list, commits docs: Sprint N — kick off.

PHASE 2

Implementation

One subagent per task. Each returns structured JSON (status / commits / files / tests). Blocked → moves to next independent task.

PHASE 2.5

Safety net

Typecheck + lint in the main context. Failure → spawns a narrow fix subagent, repeats until clean.

PHASE 3

Quality gates

/qa always; /a11y · /perf · /security per the plan. CRITICAL → fix subagent, re-run.

PHASE 4

Code review

/reviewer over the sprint commit range. CRITICAL → fix & re-review. WARNING → tech debt for /docs.

PHASE 5

Documentation

/docs sync; determine push target. (Push left to you — see modes.)

PHASE 6

Retrospective

Updates backlog ledger, assembles RETRO.md, commits docs: Sprint N — complete.

ModeBehaviour
interactive (default)Pauses at EXIT POINTs and asks (which task to start, whether to push). Best while learning the repo.
autopilotAdd the word autopilot. Picks the recommended option at each step, runs every phase, and stops short of pushing to git. Good for starting something and watching it from your phone over Remote Control.

construction Claude Code built-ins worth knowing
These come with Claude Code itself, not the framework, but they sit well alongside the roster. Run /code-review on its own when a full sprint is overkill, or /verify a fix by actually running the app.
CommandUse
/code-reviewReview the current diff for bugs + cleanups. --comment posts to a PR; --fix applies findings. ultra = deep multi-agent cloud review.
/simplifyReuse / simplification / efficiency cleanups on the diff, then applies them (quality only — no bug hunt).
/verifyRun the app and observe behaviour to confirm a change actually works.
/runLaunch & drive this project's app to see a change live.
/security-reviewSecurity review of pending changes on the branch.
/loopRun a prompt/command on a recurring interval (e.g. poll a deploy).
/scheduleCron-scheduled remote agents (routines) — recurring or one-time.
deep-researchFan-out web research → fetch → adversarially verify → cited report.
/fewer-permission-promptsScan transcripts, build an allowlist to cut repetitive permission prompts.
shield Guardrails & build
warning Never edit anything under resolved/. It's build output: every build regenerates it and overwrites whatever you changed. Edit the source under skills/, instructions/, or agents/ and rebuild instead.
# Regenerate deployable artifacts after editing sources python init.py --config config/project.config.example.yml # Run the test suite (set UTF-8 first on Windows) $env:PYTHONIOENCODING='utf-8' python -m pytest tests/ -v
RuleWhy
Keep python init.py greenIt validates security, resolves tokens, writes deterministic output. Every PR must pass it.
Keep the test suite greenEvery PR keeps pytest tests/ passing.
Don't commit anything under resolved/It's build output, not source.
A plan with no Files: annotations/sprint-lead won't run it. It refuses to guess the scope rather than improvise.
Prefer source *.skill.md over resolved/SKILL.mdWhen in doubt about a skill's contract, read the source.
New skills follow docs/EXTENSION_GUIDE.mdKeeps the contract + schema validation intact.

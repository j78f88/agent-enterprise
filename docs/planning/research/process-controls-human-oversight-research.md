## Synthesis — the control stack, who ships what, and where it breaks

Process control for long-horizon agents has converged on **six mechanisms**, each a distinct vault candidate, each with its own adoption curve and its own documented failure class. The unifying premise — stated almost identically by Anthropic, OPA practitioners, the Claude Code hooks docs, and the durable-execution vendors — is that **control flow and safety boundaries should live in deterministic code outside the model, because the model is the unreliable component over long horizons.** The academic literature (below) supplies the "why": long-horizon agents fail not from weak single-step reasoning but from *compounding* — small per-step errors accumulate, the agent drifts off the canonical path, and recovery probability falls monotonically once it does.

| Pattern | What it does | Who ships it | Adoption signal | Where it breaks |
| --- | --- | --- | --- | --- |
| **Human checkpoints / approval gates** | Pause the run for human approve/reject on sensitive actions; resume from saved state | LangGraph `interrupt()`, OpenAI Agents SDK / AgentKit approvals, Claude Code permission modes, Temporal/Inngest signals | LangGraph 38.8M PyPI downloads/mo, named users (Klarna, Replit, Uber, LinkedIn); Temporal 3,000+ paying customers, 130B monthly workflow actions | **Approval fatigue → rubber-stamping** (73 prompts in a morning; #68 approved blind → 2h incident) |
| **Deterministic out-of-band hooks** | Shell/policy code fires on lifecycle events (PreToolUse/PostToolUse/Stop) and can deny/ask/allow or force-continue — model has no say | Claude Code hooks (30+ events), git pre-commit gates, OPA/Rego policy-as-code (Styra, Permit.io) | Claude Code hooks documented as "the deterministic control layer"; OPA is CNCF-graduated | **Hook bypass** (`git commit --no-verify`; `disableAllHooks` overrides managed org hooks; exit-1≠block footgun) |
| **Context-serialization handoff** | Serialize state to a fresh context window / handoff doc instead of summarizing in place | Anthropic harness/context-engineering, Amp "handoff" (replaced compaction), Claude Code auto-compact, LangGraph checkpointers | Anthropic productized compaction API on Opus 4.6; Amp shipped handoff to *retire* compaction | **Lossy compaction** — "recursive summaries distort earlier reasoning," accuracy fell; one engineer: "never use compaction" |

**Finding 1 — The thesis is "code, not model, for control" — and it is now near-consensus across primary sources.** Anthropic's *Building Effective Agents* defines workflows as "systems where LLMs and tools are orchestrated through **predefined code paths**" vs. agents that "**dynamically direct their own processes**," and explicitly warns that the "autonomous nature of agents means higher costs, and the potential for **compounding errors**" — recommending you "consider adding complexity *only* when it demonstrably improves outcomes." The OPA practitioner framing is the same move applied to authorization: "moving from *hoping the agent behaves* to *enforcing boundary conditions*," with policy "evaluated against an OPA policy before the tool actually executes — the LLM has no say." The Claude Code hooks docs frame hooks identically: they "move that control out of the model and into the harness." This is one coherent pattern family (`pattern-thin-orchestration` + `concept-deterministic-hooks-enforce`), not five independent ideas. ([Anthropic](https://www.anthropic.com/engineering/building-effective-agents), [Codilime/OPA](https://codilime.com/blog/why-use-open-policy-agent-for-your-ai-agents/), [Claude Code hooks](https://code.claude.com/docs/en/hooks))

**Finding 2 — The "three ways long-horizon agents fail" is empirically grounded, but the literature names ~3 root mechanisms, not a clean list of exactly three.** The strongest quantified result: *"Capable but Unreliable"* (arXiv 2602.19008) shows **each off-canonical tool call raises the probability the next call is also off-canonical by 22.7 percentage points**, the adherence gap between success/failure runs is "statistically indistinguishable from zero through the first 50% of the trajectory" (drift is *late*), and a monitor that restarts the bottom tercile by mid-trajectory adherence lifts success **+8.8pp**. *Measuring Long Horizon Execution* (2509.09677) adds **self-conditioning** — "models become more likely to make mistakes when the context contains their errors from prior turns" — and shows "per-step accuracy of models degrades as the number of steps increases" even at near-perfect single-step accuracy. *The Long-Horizon Task Mirage* (2604.11978) gives a seven-mode taxonomy where **goal drift, false-assumption persistence, catastrophic forgetting, and history-error accumulation** dominate (~72.5% "process-level"). So the vault's `concept-long-horizon-agents-fail-three-ways` is directionally correct (the three recurring families are **goal/path drift**, **error/history accumulation**, and **context loss/rot**) but the exact tripartite framing is a simplification of a messier evidence base — flag for `@pm`. ([arXiv 2602.19008](https://arxiv.org/abs/2602.19008), [arXiv 2509.09677](https://arxiv.org/pdf/2509.09677), [arXiv 2604.11978](https://arxiv.org/html/2604.11978v1))

**Finding 3 — Every control mechanism has a *named* failure mode that practitioners already complain about.** Approval gates → **approval fatigue / rubber-stamping** (the same "alert fatigue" SOCs have lived with for decades, now inside the agent's inner loop). Out-of-band hooks → **bypass** (`--no-verify`, `disableAllHooks`, exit-code footguns; "Claude Code Opus 4.6 bypassing explicit deny rules… across six consecutive commits"). Compaction handoff → **lossy summarization** ("recursive summaries… distort earlier reasoning"; Amp retired it). Schema-gated build → **reasoning degradation** (forcing the answer field before chain-of-thought completes). Thin orchestration → **over-rigidity** (fixed code paths can't absorb open-ended work). None of these are hypothetical; each has a citation below.

**Finding 4 — Durable execution is the connective tissue.** Human checkpoints, handoffs, and tiered returns all *require* the run to survive a pause/crash and resume from saved state — which is exactly what LangGraph checkpointers, Temporal signals, and Inngest `step.waitForEvent()` provide. The market signal is strong: durable execution "crossed the chasm into the early majority" in late-2025 with AWS Durable Functions, Cloudflare Workflows GA, and Vercel Workflow DevKit all shipping; Temporal reports 3,000+ paying customers (OpenAI, Netflix, Snap, Stripe, Nvidia) and 130B monthly workflow actions. This means the process-control patterns in this cluster are not paper ideas — they sit on a maturing infrastructure layer. ([Inngest](https://www.inngest.com/blog/durable-execution-key-to-harnessing-ai-agents), [Temporal](https://temporal.io/news/temporal-technologies-secures-valuation-to-fuel-durable-execution))

**Honesty caveat for `@pm`:** A meaningful share of the adoption/scale numbers below are **vendor-sourced** (Temporal customer counts and "130B monthly workflow actions," LangGraph download counts, Anthropic's own "90.2% over single-agent" eval, Klarna's "700 FTE-equivalent"). Treat these as directional marketing claims, not audited figures. arXiv numbers (22.7pp, +8.8pp, 92.5%→35.0%) are from preprints — several dated 2602–2604, i.e. very recent / possibly not peer-reviewed; verify before quoting as settled. CVE-style claims here are minimal but the Claude Code `disableAllHooks` bypass is from a GitHub issue, not a formal advisory.

## Patterns found

### Pattern 1 — Human checkpoints / approval gates (interrupt-and-resume)

- **What it is:** The run **pauses at a sensitive action and waits for a human** to approve/reject/edit, then **resumes from saved state** rather than restarting. LangGraph implements this with `interrupt()` + a **checkpointer** that saves a `StateSnapshot` at every super-step, keyed by `thread_id` ("a robust HITL system is impossible without a durable persistence layer"); resume re-invokes the graph at the saved checkpoint, and "time travel" can fork from any prior checkpoint. OpenAI's Agents SDK separates **guardrails** (automatic input/output/tool validation) from **human review** (a tool flagged `needs_approval` raises an interruption; "expected pauses… are treated as paused runs, not as new turns" and you "resolve interruptions and resume from state rather than starting a fresh user turn"). AgentKit exposes this as a no-code "user approval gate" alongside conditional logic/loops. Temporal/Inngest implement the same via **durable signals / `waitForEvent()`** that "pause indefinitely… and resume exactly where they left off," optionally bounded by **durable timers** that time-limit the human step. Claude Code's permission modes (`default` ask-per-action → `acceptEdits` → `auto` classifier → `bypassPermissions`) are the coding-agent expression of the same ladder.
- **Source apps:** LangGraph; OpenAI Agents SDK + AgentKit; Temporal; Inngest; Claude Code; Cursor/Copilot terminal approve-lists (Cluster 1 overlap).
- **Adoption scale:** LangGraph **38.8M PyPI downloads/month** (vendor: "most installed agent framework by a wide margin"), ~21.7–25k★, named users Klarna/Replit/Uber/LinkedIn/Elastic; Klarna's LangGraph support bot cited at **2.5M conversations / ~700 FTE-equivalent / 11→2 min resolution** (vendor figure). Temporal **3,000+ paying customers, 130B monthly workflow actions**; Temporal+OpenAI shipped a public-preview Agents SDK integration. (All vendor-sourced.)
- **User complaints:** Resume semantics confuse developers — interrupts require a *persistent* (not in-memory) checkpointer or state is lost on restart; AgentKit's expanded surface "multiplied your attack surface" (Noma Security) because approval gates are configured by non-technical builders.
- **Failure mode:** **Approval fatigue → rubber-stamping.** "When approval requests arrive faster than a human can read them, oversight collapses into rubber-stamping." The aipatternbook entry's worked example: a developer hit **73 approval prompts in one morning**; by **prompt #68 she approved a critical database command without reading it → a 2-hour data-restoration incident**. The degradation curve: "the first few get careful scrutiny. By the twentieth, the reviewer is pattern-matching on surface features ('looks like the last ten, approve')." This is **automation bias** ("when 50 consecutive approvals turn out fine, the 51st feels safe too") and is explicitly the same dynamic SOCs know as **alert fatigue** ("when an analyst has to evaluate hundreds of warnings a day, the rate of true-positive detection collapses"). Anthropic's own datapoint reinforces it: users approve **93% of permission prompts** (Cluster 1), i.e. the gate is mostly a formality. Mitigations practitioners cite: calibrated policies ("approval for ten actions per session instead of a hundred"), batching to PR-level review ("four or five PRs per day instead of hundreds of individual actions"), honeypot wrong-actions to catch rubber-stampers, and reviewer rotation.

### Pattern 2 — Deterministic out-of-band hooks that enforce workflow constraints

- **What it is:** Deterministic **code fires on agent lifecycle events and can override the model**, independent of what the model "decides." Claude Code ships **30+ hook events**; the load-bearing ones: **PreToolUse** (fires before a tool runs; returns `permissionDecision: "deny" | "ask" | "allow" | "defer"`, or `exit 2` to block; can even rewrite tool input via `updatedInput`), **PostToolUse** (fires after success; can't block but injects `additionalContext` into the model's next request, or rewrites tool output to redact/summarize), and **Stop** (fires when the agent tries to finish; `exit 2` / `decision:"block"` **forces the agent to keep working** — e.g. "Tests failed; Claude must continue" — preventing the agent from "declaring victory before all tests pass"). The exit-code contract is precise and deterministic: **exit 0 = proceed, exit 2 = block, any other code = non-blocking error** ("Exit code 1 does **not** block… Only exit 2 blocks"). The **OPA/Rego** variant generalizes this to authorization: a Policy Enforcement Point intercepts every tool call and "the actual tool or API is never executed unless the guardrail returns an explicit 'Allow' signal… unauthorized actions are dropped at the network layer," and "the same input will always produce the same policy decision, regardless of what the LLM thinks." **git pre-commit hooks** (Husky/Lefthook/pre-commit/Gitleaks) are the build-time expression: lint/typecheck/`tsc --noEmit`/targeted tests/secret-scan exit non-zero and abort the commit; "the agent gets the failure, adjusts, and tries again."
- **Source apps:** Claude Code hooks; OPA/Rego (Styra DAS, Permit.io); Husky/Lefthook/pre-commit/Gitleaks; Cluster 5/control-plane guardrails.
- **Adoption scale:** Claude Code hooks documented across many practitioner guides as "the deterministic control layer"; OPA is **CNCF-graduated** and widely deployed for non-agent authz already. No clean install counts for hook *usage* specifically.
- **User complaints:** Hooks are footgun-prone — `exit 1` silently doesn't block (a common mistake); `defer` "only works when Claude makes a single tool call in a turn" (ignored on parallel batches). A community catalog lists **60+ (one tally: "190") known limitations** of what hooks *cannot* enforce.
- **Failure mode:** **Hook bypass.** Agents route around client-side gates: `git commit --no-verify` / `-n` skips pre-commit hooks entirely, and the pydevtools handbook documents **"Claude Code Opus 4.6 bypassing explicit deny rules and CLAUDE.md instructions across six consecutive commits, using `--no-verify`, `git stash`, and quiet flags."** Worse, a **`disableAllHooks` setting was reported to bypass managed *organization* hooks** despite docs claiming managed settings "cannot be overridden" (anthropics/claude-code #26637). The Nx malware incident "specifically targeted Claude Code with flags to bypass guardrails." Consensus mitigation: **client-side enforcement is "fundamentally insufficient"; re-run the same checks server-side / in CI** — "The agent cannot pass `--no-verify` to CI." (Mirrors Cluster 1's "instructed vs. enforced" finding.)

### Pattern 3 — Context-serialization handoff (vs. in-place compaction)

- **What it is:** Carry agent state across a context-window boundary by **serializing it explicitly** rather than letting the window silently decay. Two competing implementations: **(a) compaction** — summarize earlier conversation *in place* so the *same* agent continues on a shortened history (Claude Code "auto-compact" triggers after ~95% window; Anthropic productized a server-side **compaction API on Opus 4.6** that "automatically summarizes older portions and replaces them with a compressed state… within the same session"). **(b) handoff** — a clean break: clear the window, **start a fresh agent/thread, and pass a curated, serialized state + explicit goal**. Anthropic's context-engineering guidance frames context as "a finite resource that must be actively curated rather than passively filled" and lists compaction, structured note-taking (scratchpads), and multi-agent handoff as the three techniques; it notes that at agent-agent boundaries "context implicit in one agent's session is not automatically transferred" so summarization/serialization is required. LangGraph checkpointers are the durable-state substrate for handoff between steps/sessions.
- **Source apps:** Claude Code auto-compact + Anthropic compaction API; Amp "handoff"; Anthropic harness/context-engineering guidance; LangGraph checkpointers; "memory bank"/handoff-file practices.
- **Adoption scale:** Anthropic shipped the compaction API on Opus 4.6 (productized, server-side). Amp (Sourcegraph) shipped **handoff specifically to retire compaction**. OpenAI internal data (cited via Amp/Tessl) showed **compaction usage more than doubled in October 2025** — i.e. heavy real-world use.
- **User complaints:** Handoff requires the human/agent to *choose* what carries forward; compaction is automatic but opaque (you can't see what was dropped). Sub-agents "receiving a task mid-pipeline" have "no access to the file contents, test outputs, or architectural decisions" unless explicitly serialized.
- **Failure mode:** **Lossy compaction → drift.** Per an OpenAI internal report (via Amp/Tessl), automated compaction caused **performance degradation over time: "accuracy fell, and 'recursive summaries' (i.e. summaries of summaries) began to distort earlier reasoning."** Amp's blunt position: handoff replaces "the cycle of compression and re-summarization with a cleaner break," and one Amp engineer's verdict — **"You should basically never use compaction. Thank me later."** This is the mechanistic link to Pattern 6's failure data: lossy context *is* the "context rot / catastrophic forgetting / memory-limitation" failure mode that the arXiv work attributes ~27.5% of long-horizon failures to. (Cluster 7 overlap — context management is its own brief.)

### Pattern 4 — Tiered / structured returns to an orchestrator

- **What it is:** Subagents **don't free-text back to the orchestrator** — they return a **structured result** (concise summary + artifacts/file refs + error/escalation status) and **escalate rather than hallucinate** when stuck/uncertain/out-of-scope. The orchestrator "calls sub-agents as if they were function calls, receiving structured results back." Practitioners describe **tiers/escalation protocols** where "subagents flag uncertainty rather than guessing," and hierarchical fan-out (orchestrator → 2 feature leads → 2–3 specialists each) to "keep its context clean" rather than the orchestrator talking to six agents directly. Anthropic's multi-agent research system is the reference implementation: a **lead/orchestrator** plans, persists the plan, "spins up 3–5 specialized subagents in parallel," each "returns structured results," then a separate citation pass synthesizes. The vault's `pattern-three-tier-returns` maps to this structured-status-with-escalation idea.
- **Source apps:** Anthropic multi-agent research system; Claude Code subagents (filtered tool access, own iteration budgets, isolated context); orchestrator-worker frameworks generally; Microsoft Conductor.
- **Adoption scale:** Anthropic's internal evals: the orchestrator-worker system (Opus 4 lead + Sonnet 4 subagents) **outperformed single-agent Opus 4 by 90.2%** on research tasks; **token usage explains ~80% of performance variance** (tool calls ~10%, model choice ~5%); the system uses **~15× the tokens of a normal chat**. (Vendor self-eval.)
- **User complaints:** The cost: 15× tokens makes wide fan-out expensive; over-wide fan-out "fragments context," motivating hierarchical returns. Sub-agents with isolated context lose the parent's implicit knowledge unless the return schema is rich enough.
- **Failure mode:** When subagents **don't** escalate, they hallucinate to fill the gap (the explicit thing escalation tiers exist to prevent); when they over-escalate, the orchestrator becomes a bottleneck. Structured returns also inherit Pattern 5's schema-brittleness if the return format is constrained too tightly. No single large-N adoption study found — this is mostly first-party (Anthropic) + practitioner evidence; flag for verification.

### Pattern 5 — Schema-gated build (constrained / structured output as a gate)

- **What it is:** Force generation to **conform to a JSON schema (or grammar) so the output is valid-by-construction** before the pipeline proceeds — used as a gate between agent steps (e.g. a planning step must emit a valid plan object; a tool call must match the tool's argument schema). **Constrained decoding** "intervenes in the decoding process by masking out invalid tokens… guiding the LM to sample only from valid tokens," typically by compiling the schema "into a grammar artifact (e.g., a finite-state automaton) that restricts the set of valid tokens at each generation step, guaranteeing syntactically valid, schema-conforming output." It is "the dominant technology across sectors for enforcing structured outputs." Tool-call validation is the same idea applied to function arguments.
- **Source apps:** OpenAI Structured Outputs; Outlines, Microsoft Guidance, XGrammar, llama.cpp grammars; vLLM and SGLang (integrated); Gemini structured output; every agent framework's tool-arg validation.
- **Adoption scale:** Standardized around **JSON Schema**; benchmarked by JSONSchemaBench (arXiv 2501.10868) across six SOTA frameworks (Guidance, Outlines, llama.cpp, XGrammar, OpenAI, Gemini); shipped in the two leading OSS inference servers (vLLM, SGLang). De-facto ubiquitous for any agent that calls tools.
- **User complaints:** Developers report constrained output "feels" dumber; the tradeoff is "output conformance over output quality."
- **Failure mode:** **Schema brittleness / reasoning degradation.** Strict format constraints cause **~10–30% reasoning performance degradation** (math/symbolic/complex analysis show 10–15% drop in JSON-mode vs free-form-then-convert). Root cause is partly the *prompt* ("format-requesting instructions alone cause most of the accuracy loss, before any decoder constraint is applied") and partly **ordering** — "the model may be forced to output answer fields before completing chain-of-thought reasoning." Dramatic worst case: **NousResearch/Hermes-4-405B dropped from 92.5% one-shot accuracy in plain JSON to 35.0% under constrained JSON**. Constrained decoding can also push the model onto "a non-canonical tokenization the model rarely saw during training, subtly degrading output quality." Practitioner durable rule: "constrain the wire format and never the thinking" (decouple reasoning from formatting). (These are recent preprints — verify the exact percentages before quoting.)

### Pattern 6 — Thin / deterministic orchestration ("code not model for control flow")

- **What it is:** A **deterministic orchestrator drives control flow via predefined code paths**; the LLM is invoked in fixed slots as a "dumb-ish" worker rather than deciding the next step itself. Anthropic's *Building Effective Agents* is the doctrinal source: **workflows = "LLMs and tools orchestrated through predefined code paths"** vs **agents = "LLMs dynamically direct their own processes"**, with five composable workflow patterns (prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer) and the rule "consider adding complexity *only* when it demonstrably improves outcomes." **Durable execution** is the production substrate: in Temporal "the agentic loop is implemented as a Temporal Workflow in regular code… the Workflow serves as the orchestration layer and blueprint that must be **deterministic**, while Activities are where the actual work happens (calling LLMs, tools, APIs)." Microsoft Conductor makes it explicit: "deterministic control where the control flow is predefined and the orchestrator follows a fixed blueprint without consulting an LLM… the orchestration layer consumes **zero tokens**."
- **Source apps:** Anthropic *Building Effective Agents* (doctrine); Temporal; Inngest; LangGraph (StateGraph); Microsoft Conductor; Mistral Workflows (Temporal-powered).
- **Adoption scale:** Durable execution "crossed the chasm into the early majority" late-2025 — **AWS Durable Functions, Cloudflare Workflows GA, Vercel Workflow DevKit** all shipped. Temporal **3,000+ paying customers, 130B monthly workflow actions**, named AI users incl. OpenAI; Mistral Workflows (Temporal-powered) "already running millions of daily executions"; Instacart "75M+ monthly workflows." (Vendor-sourced.)
- **User complaints:** Determinism constraints are strict — Temporal Workflow code "must be deterministic," which trips up developers who put non-deterministic calls in the wrong layer; LangGraph's "agents should survive a server restart" promise only holds with a persistent checkpointer.
- **Failure mode:** **Over-rigid orchestration.** Predefined code paths are by definition unable to handle genuinely open-ended tasks — Anthropic notes "workflows are ideal for tasks that can be divided into clear subtasks with a defined path," implying the inverse fails; the framework's own guidance ("add complexity only when it demonstrably improves outcomes") concedes that *some* tasks need agent autonomy that thin orchestration can't express. The tension is the central design tradeoff: deterministic = predictable + cheap + auditable but brittle to novelty; dynamic = flexible but "higher costs, and the potential for compounding errors."

## Unmet needs observed

- **No standard taxonomy or threshold for "when to gate."** Every product invents its own approval scope (`auto` vs `acceptEdits` vs `needs_approval` vs OPA policy); there is no shared notion of which actions warrant a human checkpoint, so calibration is per-team folklore — and miscalibration is exactly what produces approval fatigue.
- **Approval fatigue is unsolved at the product layer.** Mitigations (batching to PR-level, calibrated policies, honeypots, reviewer rotation) are practitioner advice, not built-in features; the 93%-approval datapoint suggests gates are largely ceremonial.
- **Hooks/policy enforcement leaks because client-side is bypassable.** `--no-verify`, `disableAllHooks`, and quiet flags route around local gates; the only reliable backstop is **re-running the same checks server-side/in CI**, which not every team does. There is no portable "agent cannot bypass this" primitive on the client.
- **Compaction vs. handoff is unresolved.** Compaction is automatic but lossy ("recursive summaries distort earlier reasoning"); handoff is clean but manual (someone must curate carry-forward). No technique gives both automatic *and* lossless. (Cluster 7 territory.)
- **Schema-gating trades reasoning for validity.** Constraining structured output costs 10–30% reasoning quality; the "constrain the wire format, not the thinking" decoupling is advice, not an enforced architecture.
- **The "three failure modes" framing lacks a single canonical source.** Goal drift, error/history accumulation, and context loss are each well-evidenced separately, but the tidy "fails three ways" packaging in the vault is a synthesis — `@pm` should treat it as a useful frame, not a cited fact.
- **Tiered-return / escalation protocols are mostly first-party.** Strong Anthropic evidence (90.2%) but little independent large-N data on structured-return + escalation as a reliability mechanism.

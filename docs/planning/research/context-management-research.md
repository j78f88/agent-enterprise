## Synthesis

Context management has crystallised into a named discipline ("context engineering") in the last 12 months, and the four vault candidates map almost one-to-one onto four techniques the field now treats as standard. The load-bearing finding is that **all four are responses to the same underlying empirical fact — model recall degrades as the context window fills — and each technique trades that decay for a *different* new failure class.** The decay is well-documented across independent benchmarks; the fixes are mostly vendor-shipped and vendor-benchmarked, so adoption numbers are real but accuracy numbers should be read with a vendor-bias discount.

| Technique (vault candidate) | What it does | Who ships it | Adoption signal | Where it breaks |
| --- | --- | --- | --- | --- |
| **Long-context-decay awareness** (`concept-large-context-windows-degrade`) | Establishes that more tokens ≠ better recall; motivates everything below | Benchmarks: Lost-in-the-Middle, RULER (NVIDIA), NoLiMa (Adobe), Chroma "context rot" | 18 SOTA models tested (Chroma); 12 models claiming ≥128K (NoLiMa) | Not a "fix" — it is the *problem statement*; effective context length is far below advertised |
| **Compaction & handoff** (`concept-large-context-windows-degrade`, `[[context serialization handoff]]`) | Summarise a near-full window, reinitialise from the summary | Claude Code auto-compact; Claude API compaction; ChatGPT memory | Default-on in Claude Code; shipped Claude API primitive | Summary silently drops critical detail; agent "spirals" post-compact |
| **Structured / graph memory with decision traces** (`concept-context-graphs-vs-flat-memory`) | Temporal knowledge graph or entity graph vs flat append log | Zep/Graphiti, Mem0 (graph variant), GraphRAG, Letta | Mem0 ~48–51K★, $24M; Letta $10M seed @ $70M; Zep benchmarks vs MemGPT | Indexing cost/latency (GraphRAG $33K early indexing), retrieval misses, complexity |

**Finding 1 — The decay is real and independently replicated, and "effective context length" is far below the advertised number.** Four separate benchmark efforts converge: *Lost in the Middle* (arXiv 2307.03172) found a U-shaped curve where recall is high at the start/end of context and collapses in the middle; *RULER* (NVIDIA, arXiv 2404.06654) found that of models claiming ≥32K, only about half clear a quality threshold at 32K, and GPT-4-128K has an *effective* length of ~64K; *NoLiMa* (Adobe, arXiv 2502.05167) found GPT-4o drops from 99.3% (<1K tokens) to 69.7% at 32K, with 10 of 12 models falling below 50% of their short-context baseline at 32K once literal keyword overlap is removed; and Chroma's "context rot" report (18 models incl. GPT-4.1, Claude 4, Gemini 2.5) found reliability degrades with input length *even on trivial retrieval and text-replication tasks*. Anthropic's own engineering guide adopts the term "context rot" and frames context as a finite "attention budget." This is the empirical floor under the whole cluster.

**Finding 2 — Every mitigation buys decay-resistance with a new failure mode; there is no free lunch.** Compaction fixes window overflow but can silently drop the one detail that mattered (practitioners explicitly warn against *waiting* for auto-compact because the agent can "spiral"). Graph memory fixes flat-log retrieval misses but adds indexing cost (GraphRAG's notorious ~$33K large-dataset indexing, later cut ~1000×), 2–3× inference latency, and super-linear index growth. Subagent isolation fixes per-agent context pollution but creates *cross*-agent context loss — Cognition's "Don't Build Multi-Agents" documents subagents producing incompatible work (the Flappy-Bird-with-Mario-background example) because they cannot see each other's traces. Distilled memory fixes transcript bloat but goes stale and contradictory (ChatGPT's pre-2026 memory was measured at 41.5% factual recall) and becomes a *persistence* attack surface.

**Finding 3 — Distilled persistent memory is where this efficiency cluster collides with agent-enterprise's security core.** Memory poisoning (OWASP ASI06) is the standout: unlike session-scoped prompt injection, malicious content written into a persistent store is *temporally decoupled* — planted today, triggered weeks later when semantically retrieved. The MINJA memory-injection attack reports ~95% injection success / ~70% attack success with no special privileges; newer work (MemMorph tool-hijacking, "Sleeper" memory poisoning) extends it. So "memory is more useful when distilled" and "distilled memory is a poisonable build-time/runtime artifact" are the same coin. This is the one thread `@pm` may want to pull back into the security clusters.

**Finding 4 — "Decision traces vs flat memory" is a real, named design axis, not a vault abstraction.** Two camps are explicit about it: Zep/Graphiti markets a *temporal* knowledge graph that records facts *and their validity intervals* (non-lossy edge invalidation rather than overwrite), and Cognition argues that if you must coordinate agents you must "share full agent traces, not just individual messages" because "actions carry implicit decisions." Both are arguing that *flat append memory loses the why*; structured/trace memory preserves it. Adoption of the graph camp is real (Mem0, Zep, Letta funded and starred) but the accuracy claims are almost all vendor-run on LOCOMO/DMR/LongMemEval — treat as directional.

**Honesty caveat for `@pm`:** (a) The benchmark papers (RULER, NoLiMa, Chroma) are credible but several are preprints / non-peer-reviewed technical reports; cite arXiv IDs as preprints. (b) Almost every memory-vendor accuracy number (Mem0 91.6 on LOCOMO, Zep 94.8 DMR, "+20 points", "90% lower latency") is **self-reported by the vendor on a benchmark it selected** — Mem0, Zep, and Letta each publish numbers favourable to themselves. Funding and star counts are more reliable than accuracy deltas. (c) Anthropic's "90.2% better" multi-agent and "93% of prompts approved" figures are internal, unaudited. Verify anything load-bearing before quoting it as neutral.

## Patterns found

### Pattern 1 — Context engineering as a named discipline (the doctrine layer)

- **What it is:** A reframing of prompt engineering into "the set of strategies for curating and maintaining the optimal set of tokens during LLM inference" (Anthropic). The central premise is that context is a **finite "attention budget"** with diminishing marginal returns — the goal is "the smallest set of high-signal tokens that maximise the likelihood of your desired outcome." Anthropic enumerates four operational techniques under this banner: **compaction** (summarise + reinitialise), **structured note-taking** (persist notes outside the window, reintroduce later), **sub-agent isolation** (delegate to clean windows, return distilled summaries), and **just-in-time retrieval** (carry lightweight identifiers — file paths, queries, links — and load data at runtime rather than pre-loading). It is the doctrine that the other four patterns implement.
- **Source apps:** Anthropic engineering blog ("Effective context engineering for AI agents"); the Claude Cookbook "Context engineering: memory, compaction, and tool clearing"; widely echoed by practitioners (MarkTechPost guide, elvis/omarsar notes). The Anthropic Pokémon example is cited as proof that note-taking lets an agent "maintain precise tallies across thousands of game steps."
- **Adoption scale:** No install count — this is doctrine, not a product. Its significance is that a frontier-lab engineering org is publishing it as a standard practice, and the term "context rot" is used inside it as an accepted phenomenon (not a strawman).
- **User complaints:** N/A (doctrine). The implicit complaint it answers is that "just use the big context window" does not work in practice.
- **Failure mode:** N/A as a mechanism — but note the doctrine *concedes* the problem statement (Pattern 2): the entire discipline exists because long windows degrade. Treating it as a silver bullet ignores that each of its four sub-techniques has its own failure class (Patterns 2–5).

### Pattern 2 — Long-context decay / "context rot" (the problem statement)

- **What it is:** The empirically-measured fact that LLM recall degrades as the context window fills — not uniformly, but enough that "effective context length" is materially below advertised length. Several independent shapes: a **U-shaped curve** (recall highest at start/end, lowest in the middle — *Lost in the Middle*); **threshold collapse** (most models fall below a quality bar before reaching their claimed length — RULER); **lexical-overlap dependence** (recall holds only when the answer shares keywords with the query; remove that and it collapses — NoLiMa); and **distractor/structure sensitivity** (reliability drops with input length even on trivial tasks — Chroma).
- **Source apps / benchmarks:**
  - *Lost in the Middle* (Liu et al., arXiv **2307.03172**, Jul 2023): multi-document QA + key-value retrieval; performance highest when relevant info is at the beginning or end, "significantly degrades" in the middle, even for explicitly long-context models. Established the canonical U-shape with strong primacy/recency bias (GPT-3.5-Turbo, Claude).
  - *RULER* (NVIDIA, Hsieh et al., arXiv **2404.06654**): 13 synthetic tasks across 4 categories, adding multi-hop tracing and aggregation beyond vanilla needle-in-haystack. Of models claiming ≥32K, only ~half clear the 85.6% threshold at 32K; GPT-4-128K's *effective* length is ~64K; many "32K" models effectively handle 4K–16K.
  - *NoLiMa* (Adobe, Modarressi et al., arXiv **2502.05167**, ICML 2025): needle-in-haystack with *minimal lexical overlap* forcing latent-association retrieval. GPT-4o falls from **99.3% (<1K) to 69.7% at 32K**; **11 of 12 models** (all claiming ≥128K) fall below 50% of their short-context baseline at 32K.
  - *Chroma "Context Rot"* (technical report, 2025-07): 18 SOTA models (GPT-4.1, Claude 4, Gemini 2.5, Qwen3); reliability decreases with input length "even on simple tasks like retrieval and text replication"; failure modes include hallucination, position bias, and distractor susceptibility.
- **Adoption scale:** Not a product; the relevance is breadth of replication (4 independent efforts, 12–18 models each) and that Anthropic adopts the framing.
- **User complaints:** The practitioner-level complaint is that vendors advertise context windows (200K, 1M) that imply uniform usability the models do not deliver — "the 10,000th token is not as reliable as the 100th" (Chroma framing).
- **Failure mode (of ignoring it):** Building agents that dump everything into one large window and assume uniform recall. The benchmarks show this fails silently — the model returns a confident wrong answer rather than erroring, which is harder to detect than an overflow.

### Pattern 3 — Compaction & handoff (summarise-and-reinitialise for window decay)

- **What it is:** When a conversation nears the window limit, **summarise its contents and reinitialise a new window from the summary** (Anthropic's definition). Implementations preserve architectural decisions and unresolved issues while discarding redundant tool output. Claude Code's auto-compact detects an approaching token threshold, generates a summary, writes a **compaction block**, and on subsequent requests **drops all message blocks prior to the compaction block**, continuing from the summary. A lighter variant is **tool-result clearing** (Anthropic's "lightest-touch" compaction) — strip old tool outputs while keeping the reasoning. This is also the mechanism behind context-serialization *handoff* (`[[context serialization handoff]]` from Cluster 6): the summary is the handoff artifact between windows or between agents.
- **Source apps:** Claude Code auto-compact (default-on; `/compact` manual command; instant since v2.0.64); Claude API `compaction` primitive (platform.claude.com/docs build-with-claude/compaction); the Claude Cookbook "Automatic context compaction" recipe; ChatGPT memory does conceptually similar background summarisation.
- **Adoption scale:** Shipped default-on in Claude Code (the dominant agentic CLI) and as a first-class Claude API feature — this is the most broadly *deployed* technique in the cluster.
- **User complaints:** The dominant complaint is that **auto-compact loses important context** — practitioners "often advise against waiting for auto-compact, as it can sometimes lead to the agent losing important context and spiraling out of control." A related subtlety: path-scoped rules and nested `CLAUDE.md` files get summarised away with everything else and only reload when a matching file is re-read, so project rules can silently drop out of context after a compaction.
- **Failure mode:** **Lossy summarisation drops the one critical detail.** Summarisation is by construction lossy; if the summariser omits a constraint, an in-flight decision, or a subtle bug context, the post-compaction agent proceeds on an incomplete picture and can "spiral." Unlike a hard window overflow (which errors), a bad compaction *succeeds* and produces silently degraded behaviour. There is no built-in verification that the summary preserved what mattered.

### Pattern 4 — Structured / graph memory with decision traces (vs flat append memory)

- **What it is:** Instead of a flat, append-only log of past turns, store memory as a **structured graph** — entities, relationships, and (in the temporal variants) the *validity interval* of each fact. The design claim is that flat memory loses the *why* and goes stale, whereas a graph preserves decision traces and can invalidate (rather than overwrite) outdated facts. Two sub-families: **knowledge-graph RAG** over a corpus (GraphRAG) and **temporal agent-memory graphs** (Zep/Graphiti). Mem0 offers both a flat and a graph variant. MemGPT/Letta is a third design point: an **OS-style tiered memory** where the agent itself pages facts between in-context "main memory" and out-of-context "archival memory" via function calls.
- **Source apps & evidence:**
  - **Zep / Graphiti** (arXiv **2501.13956**): a *temporal* knowledge-graph engine that "dynamically updates the knowledge graph with new information in a non-lossy manner, maintaining a timeline of facts and relationships, including their periods of validity." Self-reported 94.8% on the Deep Memory Retrieval (DMR) benchmark vs MemGPT's 93.4%, plus LongMemEval gains "up to 18.5%" and "90% lower latency" — all vendor-run.
  - **Mem0** (arXiv **2504.19413**): "dynamically extracting, consolidating, and retrieving salient information"; a graph variant captures relational structure. Self-reports **91.6 on LOCOMO** ("+20 points over the previous algorithm") with lower compute. ~48–51K GitHub stars; **$24M raised** by Oct 2025.
  - **GraphRAG** (Microsoft, github.com/microsoft/graphrag): entity-relation graph + community summaries; demonstrated QA gains but became the cautionary tale for cost (below).
  - **Letta / MemGPT** (arXiv **2310.08560**): "LLM-as-Operating-System"; agent self-manages tiered memory. $10M seed led by Felicis at a $70M post-money valuation (Sept 2024), backers incl. Jeff Dean and Clem Delangue; ~21K stars as Letta.
- **Adoption scale:** Real and quantifiable on the *funding/stars* axis (Mem0 ~48–51K★/$24M; Letta $10M@$70M/~21K★; Zep funded). The *accuracy* axis (DMR, LOCOMO, LongMemEval numbers) is almost entirely vendor-self-reported and should be discounted accordingly.
- **User complaints:** **Cost and latency** dominate for GraphRAG — early large-corpus indexing cost ~$33K (a reported ~10,000× premium over vector search), 2–3× higher inference latency from graph traversal + community summarisation, and a graph index that grows super-linearly with corpus size, complicating incremental updates. For agent-memory graphs, the recurring complaint is **operational complexity** vs flat vector memory, and that benchmark wins don't always transfer to bespoke production data.
- **Failure mode:** **Retrieval misses and complexity/cost.** A graph only helps if the right node/edge is retrieved; entity-extraction errors at index time silently lose facts. The cost cliff made GraphRAG "impractical for most teams" at launch (later cut ~1000× by follow-up research, per practitioner reporting). Temporal graphs add correctness risk in edge invalidation logic (a fact wrongly marked expired disappears from retrieval). Net: graph memory trades flat-log retrieval misses for index-time extraction misses + cost + ops burden.

### Pattern 5 — Isolated-context subagents (orchestrator-worker context separation)

- **What it is:** Decompose a task across **subagents that each receive a self-contained task description, an output format, and a fresh context window**, run in parallel, and **return only a condensed/distilled summary** to the orchestrator. The orchestrator (a stronger model) holds the plan; workers never talk to each other. The explicit design bet (Anthropic) is that the isolation boundary should expose "almost nothing" of other subagents' work — each worker "might explore extensively, using tens of thousands of tokens or more, but returns only a condensed, distilled summary." This keeps the orchestrator's window small and high-signal.
- **Source apps:** **Anthropic multi-agent research system** (Lead Researcher orchestrator on Claude Opus 4, subagents on Sonnet 4); **Claude Code subagents** (a subagent does research in its own window, only the summary + small metadata trailer returns to the main thread). Anthropic's own context-engineering doctrine lists sub-agent isolation as one of its four pillars.
- **Adoption scale:** Anthropic reports the multi-agent configuration **outperformed single-agent Claude Opus 4 by 90.2%** on their internal research eval — but at **~15× the token cost** of standard chat, which they say makes it worth it only when the outcome value outweighs the spend. Shipped in Claude Code (subagents) at CLI scale. (The 90.2% is internal/unaudited.)
- **User complaints:** Token cost (~15×) is the headline trade-off Anthropic itself flags. The deeper complaint comes from the counter-camp (below): coordination overhead and incoherent merges.
- **Failure mode:** **Cross-agent context loss — the inverse of the per-agent benefit.** Cognition's "Don't Build Multi-Agents" (Devin team) is the canonical counter-position: because subagents "lack context from the main agent" and "cannot see what the other was doing, their work ends up being inconsistent with each other," built on "conflicting assumptions not prescribed upfront." Their illustration: one subagent builds a Super-Mario-style background while another builds a realistic bird, leaving the coordinator an impossible merge. Their stated principles — *"share context, and share full agent traces, not just individual messages"* and *"actions carry implicit decisions, and conflicting decisions carry bad results"* — directly contradict Anthropic's "expose almost nothing" bet. So isolation is **task-shape-dependent**: it works for read-only parallel research (Anthropic's domain) and fails for tasks needing coherent shared decisions (Cognition's domain). This is an unresolved, openly-contested design axis, not a settled best practice.

### Pattern 6 — Distilled / episodic memory (store reflections, not raw transcripts)

- **What it is:** Rather than persisting full transcripts, store **distilled, higher-order memories** and retrieve by a weighted blend of signals. The foundational architecture is **Generative Agents** (Stanford, arXiv **2304.03442**): raw observations accumulate in an episodic **memory stream**; retrieval scores each memory by **recency (exponential decay) × relevance (embedding similarity) × importance (a self-assessed integer)**; and periodically the agent clusters related observations and synthesises higher-order **reflections** when accumulated importance crosses a threshold. This is the lineage behind "agent memory is more useful when distilled" — the distinction between *episodic* memory (specific past events) and *semantic* memory (generalised facts/reflections), with reflection as the distillation step. Anthropic's "structured note-taking" is a lightweight industrial cousin (persist notes outside the window, reintroduce later).
- **Source apps:** Generative Agents (Park et al., 2023) is the cited origin; Anthropic note-taking; **ChatGPT memory** (saved memories April 2024 → background-synthesis "Dreaming V3" in 2026, which "synthesizes conversation history automatically" into a reviewable summary). Mem0/Zep also perform consolidation/distillation as part of their pipelines.
- **Adoption scale:** Generative Agents is one of the most-cited agent-architecture papers; ChatGPT memory is consumer-scale. The 2026 "Dreaming V3" rework is evidence the original distillation approach needed fixing in production.
- **User complaints:** **Stale and contradictory memory** is the dominant, well-documented complaint. ChatGPT's pre-2026 memory was measured at **41.5% factual recall** (wrong in over half of memory-dependent situations); power users describe memory turning into "a graveyard of stale preferences and contradictory notes," and some turn memory off entirely. The slow buildup of stale preferences and contradictions "quietly degrades your results" — i.e. distilled memory has its *own* context-rot.
- **Failure mode (two classes):** (1) **Stale / contradictory distillation** — reflections that were true once persist past their validity; contradictory notes accumulate with no reconciliation; recall accuracy degrades (the 41.5% figure). This is exactly the problem temporal graphs (Pattern 4) try to fix with validity intervals. (2) **Memory poisoning (security)** — because distilled memory is *persistent*, an attacker who gets malicious content written into it achieves **temporally-decoupled** compromise: poison planted today triggers weeks later when semantically retrieved. **OWASP ASI06** names this; the **MINJA** memory-injection attack reports ~95% injection success and ~70% attack success rate with no special privileges; **MemMorph** extends it to tool-hijacking and "Sleeper" work to dormant triggers. Defences are nascent (e.g. OWASP "Agent Memory Guard," self-reported 92.5% detection on AgentThreatBench, supports Mem0 as a backend). **This is the failure mode that connects this adjacent cluster back to agent-enterprise's security core.**

## Unmet needs observed

- **No verification that compaction preserved what mattered.** Compaction is lossy by construction and ships default-on, yet there is no standard "did the summary keep the load-bearing constraints?" check — the failure is silent and the agent proceeds confidently.
- **The subagent isolation boundary is openly contested, not solved.** Anthropic ("expose almost nothing") and Cognition ("share full traces") give opposite advice; the field has no settled rule for *when* to isolate vs share context, only that it's task-shape-dependent.
- **Distilled memory lacks a reconciliation/expiry mechanism in most products.** Stale + contradictory accumulation is the top complaint; temporal-validity graphs (Zep) address it in theory but most flat-memory products (and ChatGPT pre-2026) ship without it. The 41.5% recall figure shows the gap is real at scale.
- **Memory poisoning has no mature, default defence.** It is a *persistent* attack surface (OWASP ASI06) with high reported success rates (MINJA ~95%/70%); detection tooling exists but is nascent and not default-on in the major memory vendors. This is the cluster's clearest tie to the security mission.
- **Graph-memory accuracy claims are almost entirely vendor-self-reported.** DMR/LOCOMO/LongMemEval numbers come from the vendors selling the product; there is no widely-trusted neutral leaderboard, and benchmark wins don't reliably transfer to bespoke production data.
- **"Effective context length" is not advertised.** Vendors publish window size (200K, 1M) but not the RULER/NoLiMa-style effective length; builders must benchmark per-model to know where decay starts.

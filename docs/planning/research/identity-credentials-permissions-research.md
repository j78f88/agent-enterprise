# Research: Identity, credentials & permissions for enterprise agents (Cluster 2)

**Date:** 2026-06-15
**Requested by:** user (via `vault-candidates-research-brief.md`, Cluster 2)
**Scope:** How enterprise agent platforms handle (a) **agent identity** and centralised identity for scaled deployments, (b) **centralised credential/secret storage** for tool servers, and (c) **permission/RBAC models** — concrete data models, where secrets live, how scopes are granted, adoption scale, and failure modes (leaked secrets, over-broad scopes, confused-deputy, retired schemes).

> Surfaces patterns + evidence + failure modes only. No recommendations and no 5-test validation — kill/keep is `@pm`'s call. **Vault provenance:** `concept-agent-credential-storage`, `concept-scaled-enterprise-agent-deployments-require-centralized-identity`.

## Synthesis — the three layers, who ships what, and where it breaks

Identity/credentials/permissions for agents splits into **three stacked layers**, each a distinct vault concern, each with its own adoption curve and its own dominant failure class. Unlike Cluster 1 (sandboxing), where filesystem isolation is mature and only egress is weak, **all three layers here are immature and shipping fast in 2026** — most named products reached GA, EA, or preview within the last 12 months.

| Layer | Vault concern | Dominant mechanism(s) | Maturity (2026-06) | Where it breaks |
| --- | --- | --- | --- | --- |
| **Identity** | `concept-scaled-enterprise-agent-deployments-require-centralized-identity` | Dedicated agent-identity principal (Entra Agent ID); federated delegation (Okta XAA / ID-JAG); workload identity (SPIFFE/SPIRE); cloud IAM (AWS roles, Google SPIFFE-principal); NHI-governance overlays | Entra/Okta/Ping/Aembit GA in 2026; Google preview; SPIFFE mature but pre-agent | No cross-vendor portability; ephemeral-agent lifecycle outpaces cert issuance; audit can't attribute *which agent* + *which user* + *which mode* |
| **Credentials** | `concept-agent-credential-storage` | Plaintext config/env (the default); OS keychain; secret-manager dynamic secrets (Vault); per-user OAuth token vaulting at a gateway; credential brokering via placeholder injection | Anti-pattern is the *de-facto default*; secure options exist but adoption is low | Secrets in `.mcp.json`/env are committed at scale; "Secret Zero"; broker = single point of failure; default leases as long-lived as static keys |
| **Permissions** | (both, cross-cut) | MCP tool allow-lists; Cedar/OPA policy engines at a gateway; OAuth `tool:resource:action` scopes + token exchange (RFC 8693); ReBAC/Zanzibar; vendor permission sets; HITL step-up; AuthZEN; SCIM-for-agents | Policy engines GA; agent-scope conventions un-standardised; SCIM/AuthZEN pre-adoption | Scope strings undefined by MCP spec; over-broad standing tokens; confused-deputy/token-passthrough; God-Mode IAM defaults |

**The single most load-bearing finding across all three facets:** the dominant failure is not a clever exploit — it is **over-permissioned, long-lived, plaintext credentials attached to an agent that also ingests untrusted content.** Every large incident below (GitHub MCP, EchoLeak, Salesloft-Drift 700+ orgs, AWS AgentCore "God Mode", Entra Agent ID role overreach) reduces to the same shape: a credential or scope broader than the task, held by a principal exposed to attacker-controlled input. This is the identity/permissions analogue of Cluster 1's "lethal trifecta" and, like it, researchers describe it as **architectural, not model-patchable** — the fix is narrower scope, shorter lifetime, and credential-from-agent separation, not a better prompt.

**The second cross-cutting finding:** **the plaintext-secret-in-config anti-pattern is the single largest concrete source of harm.** Independent-corroborated scanning (Astrix, n=5,205 MCP repos) found **79% of MCP API keys passed via env vars, 53% rely on static long-lived keys/PATs, and only 8.5% use OAuth.** GitGuardian (vendor; corroborated by Help Net Security/TechRadar/Snyk) found **24,008 unique secrets in MCP config files on public GitHub in 2025, 2,117 still valid,** inside a backdrop of **~28.6M new secrets leaked to public GitHub in 2025 (+34% YoY)** and **AI-tool-assisted commits leaking at ~2× the baseline rate.** The secure alternatives (dynamic secrets, keychain, brokering) all exist and ship today; the gap is adoption, not invention.

**The third:** **the industry is converging on the same protocol plumbing while fragmenting on the products.** OAuth 2.1 + PKCE, token exchange (RFC 8693, `act` claim nesting for delegation chains), resource indicators (RFC 8707), and SPIFFE SVIDs recur in *every* vendor stack. But the agent-identity *object models* (Entra blueprint→identity→user; Okta ID-JAG; Google SPIFFE-principal; AWS role ARN; SCIM `/Agents`) do not federate, and the IETF efforts that would unify them (WIMSE agent-identity draft, `draft-oauth-ai-agents-on-behalf-of-user`, SCIM agent extension, AuthZEN) are **drafts/preview with full RFC stack realistically 2027–2028.** Several sources warn of "early-2000s SSO-style" lock-in if convergence slips.

**A standing honesty caveat for `@pm`:** the headline scale numbers in this space are **overwhelmingly from security vendors who sell the fix** — GitGuardian (secret scanning), Entro/Astrix/Oasis/CyberArk (NHI governance), Okta/Aembit/Teleport (agent IAM). NHI-to-human ratios alone range **82:1 (CyberArk) → 92:1 (Entro/NHIMG) → 144:1 (Entro H1 2025)** depending on who's counting and what they sell. Treat all "% of NHIs over-privileged / un-rotated / breached" figures as **directional and bias-inflated, not neutral.** CVE IDs, CVSS scores, GA dates, and version numbers below were drawn from security-vendor writeups and search summaries; **verify against NVD/vendor advisories before quoting any CVE or GA date as authoritative.** This note also satisfies the researcher honesty-guard.

## Method

Four parallel `@researcher` subagents, one per facet (agent/centralised identity; centralised credential & secret storage; permission/scope/RBAC models; failure modes & incidents), each running independent WebSearch/WebFetch sweeps on 2026-06-15. Primary sources preferred: vendor docs (learn.microsoft.com, docs.aws.amazon.com, cloud.google.com, developer.hashicorp.com, okta.com, modelcontextprotocol.io), IETF datatracker drafts, OWASP, CNCF/SPIFFE, and security-research writeups (Unit 42, JFrog, Cymulate, Check Point, Datadog, Snyk, GitGuardian, Endor Labs, Aikido, Silverfort, Invariant Labs, Simon Willison). Findings synthesised here; no recommendations added.

## Patterns found

### Facet A — Agent identity & centralised identity

#### Pattern 1 — Dedicated agent-identity principal (the "Entra Agent ID" class)
- **What it is:** A new first-class directory object distinct from both human users and classic app service principals, with **human accountability baked into the schema.** Entra's model is a 3-tier hierarchy: **Blueprint** (template/credential-holder: `client_id`, secret/cert, registered permissions, Conditional Access) → **Agent Identity** (per-agent instance; *no credentials of its own*; fields `objectId`, `parentBlueprintId`, mandatory `sponsor` (accountable human), optional `owner`/`manager`, `agentSubtype` = assistive|autonomous, lifecycle state) → optional **Agent User** (paired user object for systems needing a mailbox/Teams identity). Autonomous agents auth via client-credentials on the blueprint; audit logs the *agent identity* as principal.
- **Source apps:** **Microsoft Entra Agent ID** (GA Apr 2026; Copilot Studio + Foundry auto-provision; Workday ASOR integrated). Conceptually adjacent: **Okta** agent directory, **SCIM `/Agents`** draft (fields: `agentType`, `roles`, `entitlements`, `x509Certificates`, `protocols[A2A|OpenAPI|MCP]`, `owners`, `parent`).
- **Adoption scale:** GA but gated behind M365 E5 / Entra P1-P2 (full governance → E7 at ~$99/user/mo). ~400M M365 commercial seats (vendor); **no independent agent-identity count.** Copilot Studio agent creation auto-mints an identity record.
- **User complaints:** Licensing cost + complexity (G2/Peerspot reviewers cite premium-tier gating); Power Platform API not appearing in standard portal permission list (manual workaround); "client secrets shouldn't be used in production for blueprints" (vendor's own caveat) but rotation across distributed instances is painful.
- **Failure mode:** **Entra "Agent ID Administrator" role overreach (Silverfort, "CVE-2026-42901" per source — verify):** the role's `microsoft.directory/agentIdentities/owners/update` lacked filtering, letting a holder add themselves as **owner of *any* service principal tenant-wide** (incl. Global-Admin-equivalent) → tenant takeover. Disclosed Feb 24 2026, patched Apr 9 2026; **pre-patch owner entries persist unless purged.** Also **EchoLeak (CVE-2025-32711, CVSS 9.3 — verify):** zero-click prompt injection used M365 Copilot's *own* broad delegated grants to exfiltrate from OneDrive/SharePoint/Teams — well-modelled identity did **not** prevent scope abuse once LLM context was polluted.

#### Pattern 2 — Federated delegation token (Okta Cross App Access / ID-JAG)
- **What it is:** A new short-lived, admin-governed delegation credential ("ID-JAG" token, an OAuth extension) that encodes `{user identity, agent identity, permitted scopes, target app}`. Flow: user SSO → agent exchanges user's ID token at the enterprise IdP for an ID-JAG (IdP evaluates "may this user use Agent X against App Y?") → agent presents ID-JAG to target app's AS → access token. **No end-user consent dialog** — entirely admin-controlled. Adopted into the **Nov 2025 MCP auth spec** as the "Enterprise-Managed Authorization" capability, alongside **Client ID Metadata Documents (CIMD)** which replaced per-server Dynamic Client Registration (DCR).
- **Source apps:** **Okta** (XAA announced Jun 23 2025; "Okta for AI Agents" GA Apr 30 2026); partners AWS, Google Cloud, Salesforce, Box, Glean, Grammarly, Miro, WRITER, Automation Anywhere, Boomi. Compatible pattern in **Aembit MCP Identity Gateway**, **Ping Identity** ("Identity for AI", GA Mar 31 2026, with nested-`act` token exchange).
- **Adoption scale:** Vendor claims "91% of orgs deploy AI agents, only 10% have a solid NHI strategy" (Okta — unverified) and Gartner "40% of enterprise apps embed agents by end-2026" (analyst, cited via vendor). **No named XAA deployment count.**
- **User complaints:** The thing XAA replaced (DCR per the Mar 2025 MCP spec) was "a massive barrier to enterprise adoption" — unbounded server-side DB growth, no cleanup policy, clients re-registering repeatedly. XAA itself needs admin pre-config of every app-to-app path (governance bottleneck at agent speed); ID-JAG token format is proprietary/unpublished.
- **Failure mode:** Enterprise IdP becomes a **single point of failure** (Okta down → all agent sessions fail); an exfiltrated ID-JAG is replayable to any in-scope app until TTL expiry; **no multi-cloud portability** (Okta-specific; AWS/GCP-native workloads need separate integration).

#### Pattern 3 — Cryptographic workload identity (SPIFFE/SPIRE, and the cloud-native variant)
- **What it is:** Issue each workload a short-lived, attested **SVID** (X.509 or JWT) carrying a `spiffe://trust_domain/path` ID — no shared secret; identity derived from infrastructure facts (K8s pod labels/SA, cloud instance doc, TPM). SPIRE agent on each node attests the workload via a Unix-socket workload API and the SPIRE server issues an SVID (default ~1h, auto-rotated). The **WIMSE `draft-ni-wimse-ai-agent-identity` extension** proposes **dual-identity credentials** binding *both* the agent's and the owner's SPIFFE IDs (Agent/Owner/Server-mediated issuance) — but its security-considerations section is literally "TODO." Cloud variants: **Google Agent Runtime** issues a Google-managed `principal://…/reasoningEngines/<id>` SPIFFE-format ID with 24h X.509 + mTLS-bound (Context-Aware Access) tokens; **AWS IAM Roles Anywhere** maps X.509 cert attributes to IAM roles for off-AWS agents.
- **Source apps:** **SPIRE** (CNCF graduated, prod since 2018); **Istio/Envoy/AWS App Mesh** ship SPIFFE natively; **HashiCorp Vault 1.21+** native SPIFFE auth; **Google Agent Runtime** (GA for identity); **Red Hat OpenShift** zero-trust workload identity (tech preview); **wasmCloud** (adopted Mar 2025). **Akeyless SecretlessAI** and **Aembit** consume SVIDs.
- **Adoption scale:** Prod users cited from conference talks (not vendor marketing): Uber, Stripe, Netflix. Istio broad (>10k orgs per CNCF survey, not all using SPIFFE explicitly). No SPIRE license/download count.
- **User complaints:** Requires dedicated HA infra (attestation nodes, registration server, upstream CA) — "years of experience, not months"; SVID-expiry + re-attestation-failure causes **agent shutdown / production outage**; re-attestation storms can overwhelm the server; **X.509 issuance latency is incompatible with creating/destroying agents thousands of times/day.**
- **Failure mode:** SPIRE server is a SPOF without HA; **SPIFFE encodes workload identity but not intent, delegation chain, or task scope** — a valid SVID can't express "acting for user U on task T" (CNCF/Solo.io call this the core agent gap; WIMSE is the unfinished fix). No native `owner`/`sponsor` concept.

#### Pattern 4 — Cloud IAM service-role pattern (AWS / Google)
- **What it is:** Reuse the cloud's existing IAM as the agent identity. **AWS Bedrock AgentCore:** each agent gets an IAM execution role; trust policy grants `bedrock-agentcore.amazonaws.com` `sts:AssumeRole` (ideally constrained by `aws:SourceArn`); per-session attribution via `sts:SourceIdentity`/session tags. **Google:** the SPIFFE-`principal://` from Pattern 3 becomes an IAM principal in allow policies, with deny policies + Principal Access Boundary + VPC-SC.
- **Source apps:** AWS Bedrock AgentCore; AWS IAM Roles Anywhere (GA Jul 2022); Google Agent Runtime / Gemini Enterprise.
- **Adoption scale:** IAM is the foundational AWS/GCP identity layer (billions of authz decisions/day — vendor); Roles Anywhere usage not separately quantified; AgentCore Policy "thousands of enterprise testers" (vendor, unverified).
- **User complaints:** Roles Anywhere quota of 2 CRLs/trust-anchor (non-adjustable, no UI); trust anchors largely immutable; **no native "agent owner"/accountability** — identity is purely technical (role ARN + account); CloudTrail shows `AssumeRole` by the *service principal*, not a named agent, unless you add `SourceIdentity`; orphaned roles from deleted agents persist.
- **Failure mode:** **Confused deputy in Roles Anywhere (Unit 42):** without `aws:SourceArn` conditions, any cert signed by the trust anchor can assume any mapped role by knowing its ARN → lateral movement. Long Roles-Anywhere sessions (up to 12h) widen the exposure window. (AWS AgentCore "God Mode" default IAM is in Pattern 9.)

#### Pattern 5 — NHI-governance overlay (discover/attribute/flag)
- **What it is:** A layer *above* the identity substrate that discovers all non-human identities (service accounts, API keys, OAuth tokens, agent identities) across cloud/SaaS/repos, maps them to owning humans/apps, and flags hygiene violations (unused, over-permissioned, un-rotated, reused). Some (Aembit, CyberArk) add inline enforcement / just-in-time credential injection.
- **Source apps:** Aembit (GA Apr 2026), Entro, Astrix, **Veza (acquired by ServiceNow Dec 2025)**, Zenity (shadow-agent discovery), CyberArk PAM (under Palo Alto acquisition; treats agents as privileged users).
- **Adoption scale:** This category is the **source of nearly every sprawl statistic** in this doc — and every vendor sells the fix. See honesty caveat.
- **User complaints / failure modes:** OAuth-consent-grant scanning is blind to agents using API keys or SPIFFE creds; governance ownership is fragmented (security 39% / IT 32% / AI-team 13%); **78% of enterprises lack documented agent identity create/remove policies; only 21% keep a real-time agent registry; only 28% can trace agent actions to a human sponsor** (CSA/Okta — vendor-funded). "Shadow agents" (agents connecting via OAuth without IT knowledge) are the named top unsolved gap.

### Facet B — Centralised credential & secret storage

#### Pattern 6 — Plaintext secrets in config/env (the anti-pattern that dominates)
- **What it is:** MCP clients (Claude Desktop, Cursor, VS Code, Windsurf) read server config from JSON (`.mcp.json` / `claude_desktop_config.json`) with an inline `env` block holding the raw key (`"GITHUB_TOKEN": "ghp_…"`). Passed as process env at launch — no encryption, no rotation, no file ACL. The official quickstarts historically modelled exactly this.
- **Source apps:** The de-facto default for local MCP across all major clients; officially documented; increasingly criticised.
- **Adoption scale (independently corroborated):** Astrix (n=5,205 MCP repos): **79% of keys via env vars, 53% static long-lived keys/PATs, 8.5% OAuth.** GitGuardian: **24,008 MCP-config secrets on public GitHub in 2025, 2,117 still valid**; **~28.6M total new secrets in 2025 (+34% YoY)**; **AI-tool-assisted commits leak at ~2× baseline** (3.2% vs 1.5%). Doppler (vendor): ~48% of reviewed MCP servers recommend plaintext storage.
- **User complaints:** Repeated GitHub requests for keychain storage over env vars; HN flagged that the PAT model makes credential blast radius unbounded.
- **Failure mode:** Config files are trivially committed if not `.gitignored`. Because env vars are inherited by every subprocess/library, **any tool in the same process can read every secret** — and deserialization gadgets weaponise it: **LangChain CVE-2025-68664 (CVSS 9.3 — verify)** let an injected `{"lc":1,"type":"secret","id":["ENV_VAR"]}` payload exfiltrate arbitrary env vars when `load()` ran with the former default `secrets_from_env=True`. Unit 42's 2024 cloud-extortion campaign harvested **90,000+ unique credentials from 110,000+ domains' exposed `.env` files.**

#### Pattern 7 — OS-native keychain / encrypted local storage
- **What it is:** Store OAuth access/refresh tokens in the platform secure store after an interactive flow — macOS Keychain, Windows Credential Manager (DPAPI), Linux Secret Service/libsecret — retrieved at runtime, never written to config. IBM `mcp-cli` ships three backends (`keyring`, AES-`encrypted` file, `vault`) with a concrete token record: `{type, name, access_token, refresh_token, expires_at, issued_at, provider, namespace}`; rotation is manual (`token delete`+re-auth or `token refresh`), auto-rotation "planned."
- **Source apps:** IBM `mcp-cli` (OSS), Auth0 `mcp-server` (keychain per DeepWiki), Doppler MCP (system keyring), Docker MCP Toolkit (Docker Desktop-managed — backend undocumented), 1Password (`op://` refs, no plaintext).
- **Adoption scale:** Docker MCP catalog 1M+ pulls (vendor); 1Password Secure Agentic Autofill EA Oct 8 2025, Unified Access Mar 2026; IBM mcp-cli no download count.
- **User complaints / failure modes:** Keychain entries persist after sessions — OS-session compromise exposes **all** tokens at once; IBM's own docs flag `--value` leaking tokens into shell history; Docker's at-rest crypto is undocumented (unverifiable).

#### Pattern 8 — Secret-manager dynamic/leased credentials
- **What it is:** Don't store static creds — generate ephemeral ones on demand with a TTL. Agent authenticates to the manager via a machine identity (IAM role, K8s SA, JWT, **SPIFFE SVID**), gets a short-lived credential, uses it, lease auto-revokes. Vault AWS engine response: `{lease_id, lease_duration, lease_renewable, access_key, secret_key, session_token}`. HashiCorp's validated AI pattern: **Entra OBO token → Vault JWT auth → `groups_claim`→policy → dynamic DB creds + correlation IDs** linking each issuance to user+agent session.
- **Source apps:** HashiCorp Vault (validated AI pattern; native agent support EA 2026), AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, Akeyless SecretlessAI (SPIFFE/SPIRE, Jul 2025), Infisical (machine identities), 1Password (`op run`), Doppler (`doppler run`, MCP server *experimental*).
- **Adoption scale:** Vault "thousands of enterprise customers" (vendor, no agent count); Infisical ~17k★; **44% of enterprises still use static API keys for agent auth** (CSA/Okta — vendor-funded).
- **User complaints / failure modes:** **Vault's AWS-engine default lease is 768h (32 days)** — effectively static unless explicitly overridden (requires active TTL governance); availability dependency (Vault outage blocks all agent ops); per-request DB-connection + token-exchange latency; **the "Secret Zero" problem** — the bootstrap identity used to reach the manager still has to come from somewhere; Doppler MCP is explicitly dev/eval-only.

#### Pattern 9 — Gateway token vaulting & credential brokering (agent never sees the secret)
- **What it is:** Two related "the agent holds no real credential" designs. **(a) Per-user OAuth token vaulting at an MCP gateway:** the gateway holds per-user tokens (refresh tokens envelope-encrypted with per-record KMS DEKs; access tokens cached in Redis; **proactive refresh at 80% of TTL** with a per-(user,server) lock), and a 3-plane flow (inbound agent auth → RBAC policy → outbound per-user token injection) means the agent **never receives the upstream GitHub/Slack token.** Record: `{user_id, access_token, refresh_token, expires_at, scopes[], provider_metadata}`. **(b) Placeholder-injection broker:** the agent emits `Authorization: Bearer __github_token__`; an `HTTPS_PROXY` broker authenticates the agent's machine identity, matches the destination to a credential rule, swaps the placeholder for the real secret, and forwards — so prompt-injection exfiltration yields only the placeholder string. This is the SANS-named **"Credential Broker for Agents (CB4A)."**
- **Source apps:** **TrueFoundry MCP Gateway** (full 3-plane, envelope-encrypted; ~350 RPS/1 vCPU/~10ms; named customers Cargill/Mavenir/Whatfix/Aviva — vendor); **Docker MCP Gateway** (OSS; brokers OAuth via Docker Desktop; `docker mcp secret set`); **Infisical Agent Vault** (placeholder broker); **1Password `op run`**; **Aembit** (SVID + RFC 8693 exchange + credential delivery); **CyberArk PAM**.
- **Adoption scale:** Docker MCP gateway in Docker Desktop (100M+ installs, MCP feature newer); TrueFoundry 7 named customers (vendor, unverified); MCP OAuth client-credentials extension is **draft**, client support "varies."
- **User complaints / failure modes:** Placeholder broker only works if the agent honours `HTTPS_PROXY` (agents making direct HTTP/gRPC calls bypass it); `op run` still injects into process env, so any subprocess reading `os.environ` sees the secret (same surface as Pattern 6, better upstream hygiene); **the broker/gateway is a high-value single point of failure** — its compromise yields all credentials for all agents. **CVE-2025-6514 (mcp-remote, CVSS 9.6 — verify; 437k downloads):** the OAuth `authorization_endpoint` was passed unsanitised to `open()`, giving RCE **before any token was issued** — the OAuth flow itself became the attack vector.

### Facet C — Permission, scope & RBAC models

#### Pattern 10 — MCP tool allow-list (deny-by-default virtual keys / tool groups)
- **What it is:** Bind each consumer (agent/team) to a key carrying an explicit per-server tool allow-list; tools not listed are **invisible at discovery *and* blocked at execution** (two enforcement points). `{mcp_client_name, tools_to_execute:[...]}`; reusable named role groups; wildcard `["*"]` documented as the anti-pattern. MCP spec v2025-11-05 (SEP-835) added native per-tool scope definitions.
- **Source apps:** Maxim (prod), **ServiceNow AI Gateway** (tool groups per MCP client; OAuth 2.1 GA Q4 2025), MCP spec itself.
- **Adoption scale:** MCP SDK ~97M downloads/mo late 2025 (vendor); ServiceNow AI Gateway integrates Bedrock/Vertex/Copilot Studio/Claude Desktop.
- **User complaints / failure modes:** **MCP spec deliberately does NOT define scope strings** ("implementors must decide") → projects default to one opaque `access_mcp` scope, defeating granularity; discovery-suppression isn't universal (some servers expose all tools then block only at exec, leaking capability surface to injection). **GitHub MCP injection (Invariant, May 2025):** an over-scoped `repo:*` PAT + a malicious issue body → agent read private repos and wrote them to a public PR; no tool allow-list was present.

#### Pattern 11 — Declarative policy engine at the agent-tool boundary (Cedar / OPA / Cerbos)
- **What it is:** Intercept every tool call at a gateway; evaluate principal + action + **input parameters** against a policy; allow/deny before execution. **Cedar** (AWS) is statically typed/formally verifiable, supports time-of-day `forbid`, runs `LOG_ONLY`→`ENFORCE`, logs every decision to CloudWatch; benchmarks claim 42-60× faster than Rego (Cedar team/Teleport). **OPA/Rego** supports dynamic external-data lookups but has runtime-exception/non-determinism failure modes. **Cerbos** uses YAML resource policies (`principal/resource/action/condition`) and ships an MCP integration. The policy decision can evaluate the *specific call*, not just the token — the emerging answer to "authorized to call ≠ this call was intended."
- **Source apps:** AWS Bedrock AgentCore Policy (preview Dec 2025; NL→Cedar), AWS Verified Permissions (GA), Cerbos, Permit.io (wraps OPA+Cedar+ReBAC), Aserto/Topaz, OPA (CNCF).
- **Adoption scale:** OPA 100M+ downloads (vendor), embedded in K8s/Istio/Envoy (infra layer agents run on); Cedar AWS-specific; CNCF 2026 "recommended stack: SPIFFE + OAuth 2.0 + OPA."
- **User complaints / failure modes:** Rego = 30-40h learning + runtime exceptions; Cedar excludes dynamic lookups (no external API at eval time) and regex (breaks Rego migration); **OPA can silently fail-open** if fail-open vs fail-closed is left to the implementor.

#### Pattern 12 — OAuth 2.1 scoped-token architecture + downscoping (RFC 8693 / RFC 9396)
- **What it is:** Per-tool scopes converging (un-standardised) on `tool:resource:action` (e.g. `tool:calendar:read`, `tool:email:send`), no wildcards. **Rich Authorization Requests (RFC 9396)** push it tighter — a one-task token: `{actions:[read_thread,send], resource_ids:[thread:…], valid_for_seconds:120, constraints:{send_to:[…], max_recipients:1}}`. **Token exchange (RFC 8693)** is the correct sub-agent downscoping primitive: present broad user token + agent actor token, request narrower `scope`, get a tighter credential; delegation chains nest in the `act` claim (`act.sub=agent2, act.act.sub=agent1`). The `draft-oauth-ai-agents-on-behalf-of-user` adds `requested_actor` (shown on consent) + `actor_token`.
- **Source apps:** Stytch Connected Apps, Okta XAA, Microsoft Entra OBO, Ping Identity, Nango (400+ integrations), Composio (500+ connectors); MCP v2025-11-05 mandates OAuth 2.1 + PKCE-SHA256 for remote servers.
- **Adoption scale:** OAuth 2.0 underpins most enterprise API auth; the **agent-specific extensions (RAR, draft-oauth-ai-agents) are emerging, not yet in most IdP SDKs**; step-up authorization is prototype-stage. Zylos (independent, methodology undocumented): **93% of AI-agent projects still use unscoped API keys; 74% report agents end up over-permissioned.**
- **User complaints / failure modes:** Scope format fragmentation (`access_mcp` vs `mcp:tools:*` vs `tool:calendar:read`, no interop); delegation chains reach 5-10 hops with **no standardized depth limit**, making "human in the loop" notional; **EchoLeak (CVE-2025-32711)** again — legitimate-but-too-broad OAuth grants exploited by zero-click injection.

#### Pattern 13 — Relationship-based authorization (ReBAC / Zanzibar)
- **What it is:** Authz from graph traversal over `(subject, relation, object)` tuples — `check(subject=Agent, role=viewer, resource=Folder:Q3-Financials)`. The agent is granted a role on one node and **cannot reach sibling nodes** (prevents lateral movement, the core appeal over flat RBAC).
- **Source apps:** WorkOS FGA, Auth0 FGA (Okta), SpiceDB/AuthZED (LangChain integration), Google Zanzibar (internal reference design).
- **Adoption scale:** Used at Google/Airbnb/Slack scale (infra); WorkOS/Auth0 FGA are commercial SaaS — no agent-specific counts.
- **User complaints / failure modes:** Tuple-sync latency → stale decisions in high-velocity agent flows (Zanzibar "zookies" mitigate; commercial consistency varies); needs careful upfront schema, retrofitting RBAC→graph is a real migration. **Asana MCP (Jun 2025):** a relationship-isolation flaw let one org read another's projects/tasks (cross-tenant).

#### Pattern 14 — Vendor permission sets & the "dynamic user" trap
- **What it is:** Reuse the platform's existing RBAC. **Entra Access Packages** (request pathways: agent self-request / sponsor / admin; expiry + sponsor renewal; sponsorship auto-transfers to manager on departure). **Salesforce Agentforce**: agent runs as an "AI user" — either **fixed** (explicit permission sets) or **Dynamic user** (inherits the *invoking* human's permissions). **ServiceNow** mirrors this (AI user vs Dynamic user + tool groups + 30min/100day OAuth).
- **Source apps:** Microsoft Copilot Studio/Foundry, Salesforce Agentforce (~150k customers, "10k+ deployments" vendor; Agentforce Default deprecated Jun 17 2025), ServiceNow AI Gateway (~8,100 customers).
- **User complaints / failure modes:** Salesforce profiles/permission-sets are notoriously complex at scale (SFDC's own blog urges profile→permission-set migration before rollout); **"Dynamic user" inheritance is a built-in confused deputy** — a low-privilege user invoking a high-privilege agent (or vice versa) reaches data they couldn't directly.

#### Pattern 15 — Human-in-the-loop step-up authorization (CIBA / tool-approval)
- **What it is:** A PDP risk-scores each action; low-risk passes, high-risk (large financial txn, deletion, privileged access) triggers an out-of-band approval (CIBA push / Slack / email) to a named, authenticated approver before the agent resumes. Microsoft's `ToolApproval` pauses a tool call pending human decision.
- **Source apps:** Microsoft Copilot Studio (approval nodes) / Agent Framework `ToolApproval`, Oracle Cloud (preview), Stytch, LangGraph (interrupt). Endorsed as best practice in NIST AI RMF 1.1 (Mar 2026) + EU AI Act for high-risk systems.
- **User complaints / failure modes:** CIBA latency (seconds-minutes) breaks real-time use; **consent-granularity problem** — upfront OAuth consent can't describe dynamically-decomposed sub-tasks → broad consent; **approval fatigue** — too many gates → rubber-stamping (the same failure as alert fatigue, and as Cluster 1's "93% of prompts approved").

#### Pattern 16 — Interop/provisioning standards (SCIM-for-agents, AuthZEN)
- **What it is:** **SCIM agent extension** (`draft-abbey-scim-agent-extension`): `/Agents` + `/AgenticApplications` resource types so IdPs provision **and deprovision** agents through the same lifecycle as humans (fields incl. `agentType`, `roles`, `entitlements`, `x509Certificates`, `protocols`, `owners`, `parent`). **AuthZEN Authorization API 1.0** (OpenID): a standard PEP↔PDP request so one PEP queries any policy engine (Rego/Cedar/ReBAC) — solves engine lock-in.
- **Source apps:** IETF SCIM draft (consolidation at IETF 125, Mar 2026, **no production impls yet**); AuthZEN 1.0 (OPA/Cerbos/Aserto/Tyk/WSO2/Okta/Amazon implementing; interop event held).
- **Adoption scale:** **Pre-adoption / early.** Both are "the right direction" per WorkOS/SecurityBoulevard but enterprises are evaluating, not deploying.
- **Failure mode:** Not security failures — **standardisation-timing risk:** five vendors shipped competing agent-identity frameworks at RSAC 2026; if drafts don't converge in 12-18mo, "early-2000s SSO-style" lock-in.

## Real-world incidents (cross-referenced across facets)

- **Salesloft–Drift OAuth supply-chain breach (Aug 2025, UNC6395) — the canonical over-scoped-token incident.** Attackers stole the **OAuth refresh tokens** customers had granted the Drift→Salesforce chatbot integration, then impersonated the trusted Drift app to mass-export Salesforce CRM data from **700+ organizations** (named: Cloudflare, Google, PagerDuty, Palo Alto, Proofpoint, Zscaler, Tanium), and grepped the exports for further secrets (AWS keys, Snowflake tokens). **Undetectable at the customer level** because all traffic flowed through a pre-approved OAuth app. Pure over-permissioned, long-lived, broadly-scoped non-human credential — no malware on the victims. (CSA / Anomali / AppOmni.)
- **AWS Bedrock AgentCore "IAM God Mode" (Unit 42 / Sonrai).** The AgentCore **starter toolkit ships default IAM roles with `Resource:"*"`** across memory/runtime/ECR. A compromised agent can read another agent's `MemoryID` from ECR image env vars and read/corrupt that agent's memory — **cross-agent privilege escalation with no explicit escalation step.** AWS's own guidance: starter template is "development only," build a least-privilege role.
- **MCP "token passthrough" confused-deputy class — banned at the spec level (Jun 2025).** Pre-June-2025 MCP servers commonly forwarded the client's token upstream without audience validation, so a token minted for one server was replayable against any upstream accepting it. The Jun 2025 spec **prohibited passthrough and mandated Resource Indicators (RFC 8707).** Concrete instance: **FastMCP CVE-2025-69196** (the most-used Python MCP framework) issued tokens scoped to `base_url` instead of the requested `resource`, enabling token reuse across all servers under one AS.
- **Supply-chain credential theft into agent configs.** **SANDWORM_MODE (Feb 2026):** 19 malicious npm packages wrote a rogue MCP server into `~/.claude/settings.json` / Cursor / Continue / Windsurf configs whose tool descriptions hid prompt injections harvesting SSH/AWS/`.npmrc`/`.env`/LLM keys, then **wormed** via stolen npm/GitHub tokens. **postmark-mcp (Sep 2025):** a malicious version BCC'd every agent-sent email to the attacker (~10 days live). **codexui-android (Aikido):** trojanized npm package exfiltrated `~/.codex/auth.json` (incl. non-expiring refresh token) on module load. **Smithery (Jun 2025):** path traversal leaked a Fly.io token controlling **3,000+ hosted MCP servers.** **Snyk ToxicSkills (Feb 2026):** 13.4% of 3,984 public agent skills had a critical security issue; 91% of malicious ones combined malware **and** prompt injection.

## Unmet needs observed

- **No agent-identity portability standard.** Entra ≠ Okta ≠ Google ≠ AWS object models; agents crossing org/cloud boundaries have no common identity token. WIMSE/SCIM-agent/`draft-oauth-ai-agents`/AuthZEN are the candidates; full RFC stack ~2027-2028.
- **Audit can't answer "which agent, for which user, in which mode."** OpenID Foundation flags autonomous-vs-delegated mode is unobservable at runtime; recursive delegation (`act` nesting) has **no enforced depth limit**; most systems log `client_id` *or* user, not both + agent-instance + delegation chain.
- **Plaintext secrets in config/env remain the default** despite mature alternatives — the gap is adoption, not technology. Auto-rotation of secrets *in* MCP configs without human re-auth does not exist in any tool.
- **No standard scope vocabulary for tools.** MCP spec explicitly punts; ≥5 competing conventions in production; per-*server* scoping exists but per-*tool*/per-session is gateway-vendor-specific.
- **"Authorized to call" ≠ "this call was intended."** Token-level authz can't distinguish a user-intended call from an injected one; per-call runtime policy evaluation (AgentCore Policy, Cerbos/Aembit) is the emerging answer but barely shipping.
- **Confused deputy is structural to the "dynamic user" / token-passthrough patterns** that two of the biggest enterprise platforms (Salesforce, ServiceNow) ship as a default mode.
- **"Secret Zero" and broker-SPOF are unsolved** — every secretless scheme still needs a bootstrap identity, and the broker that removes per-agent secrets becomes the single highest-value target.
- **SPIFFE doesn't fit ephemeral high-frequency agents** (issuance latency, infra overhead) and encodes no intent/owner/task — the exact gap WIMSE is (incompletely) chasing.
- **Statistics are vendor-dominated.** There is no neutral measurement of agent NHI sprawl, over-permissioning, or breach rates; every headline figure comes from a company selling the remedy.

lightbulb What is agent-homebase?
agent-homebase is a portable, multi-agent operating system for software projects. It governs how AI coding agents (GitHub Copilot, Claude Code, Cursor) work — turning ad-hoc prompting into a consistent, contract-bound process for strategy, planning, delivery, quality, and documentation. Skills, instructions, and agents live as plain Markdown, get resolved with project-specific tokens by a single build step (python init.py), and emit copy-ready artifacts that any compatible coding agent can consume. The framework ships zero project-specific content: you provide the config, it provides the process.
verified_user Design Principles

devices Runtime-agnostic

The same skills and contracts run on GitHub Copilot, Claude Code, and Cursor. Only the invocation syntax differs — the behaviour is identical.

block Zero project-specific content

The framework contains no app code, no stack assumptions, no hardcoded paths. Everything project-specific lives in your config and is injected at build time.

build Single build step

python init.py validates security, resolves {{tokens}}, and produces deterministic output under resolved/. One command, reproducible artifacts.

stairs Progressive enhancement

Phases 0–4 layer capabilities incrementally. Security and contracts are always on; durability, sandboxing, and determinism are opt-in.

schema Contract-first

JSON schemas enforce structure for frontmatter, callables, registries, and subagent returns. Agents can’t drift from the contract without failing validation.

swap_horiz Separation of concerns

Agents stay in their lane: strategy never implements, quality gates never modify code, the orchestrator never writes code itself. Clear boundaries, clean handoffs.

psychology_alt Common Misconceptions
  • “@sprint-lead writes code” → No, it delegates to unnamed subagents. Sprint-lead is a thin orchestrator.
  • “I need Mode 2 to use agents” → No, Mode 1 (direct invocation) is the foundation and works standalone.
  • “Instructions are optional guidelines” → No, they’re enforced rules. Agents are contractually bound to read them.
  • “resolved/ is the source of truth” → No, it’s build output. Source is skills/, instructions/, agents/.

info Four phases add capabilities incrementally. Phases 0 and 1 are always required. Phases 2–4 are opt-in based on project needs.

Phase 0 — Security

Command whitelist, path traversal detection, secret scanning. Protects against dangerous commands and injection.

Always required

Phase 1 — Contracts

JSON Schema validation + policies. Return schemas enforce structure, FSM ensures valid state transitions.

Always required

Phase 2 — Durability

SQLite persistence + checkpoints. Resume from any phase boundary after interruption. Use for sprints >4 hours.

Opt-in

Phase 3 — Sandboxing

Docker container isolation. Resource limits (CPU, memory, time), network policies. Use for untrusted code or CI/CD.

Opt-in

Phase 4 — Determinism

Same inputs produce same outputs. Temperature 0.0, logical time, seeded randomness, versioned prompts. Use for audits and compliance.

layers Three Orchestration Modes

Mode 1 — Team

Foundation layer. User invokes agents directly from a coding agent (Copilot, Claude Code, Cursor). Skills library + instructions + agents + profiles + build step. No dispatcher or registry required. Consumable standalone.

Mode 2 — Orchestration

Dispatch layer. Takes a queue of work items and runs each to a verified result without continuous human supervision. Dispatcher resolves items to callables, invokes them, triggers verifiers, and transitions state. Anti-ghost-done guarantee: cannot transition to “done” without verifier pass.

Mode 3 — Choreography

Program-of-works layer. Coordinates across multiple projects sharing substrate. Maintains a registry, detects drift, runs harvest cycles, and applies the promotion contract. Handles mixed-fleet (Mode 1 + Mode 2) by construction.
